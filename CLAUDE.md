# Проект: Вики по продлению ВНЖ Португалии (по чату Telegram)

## Глобальная задача

Построить **вики-справочник** по продлению ВНЖ Португалии на основе экспорта
истории большой русскоязычной Telegram-группы иммигрантов. Фокус — новый
веб-портал продлений AIMA (`portal-renovacoes.aima.gov.pt`), запущенный
~11.07.2025, и весь связанный с ним опыт сообщества: этапы заявки, оплата,
документы, статьи ВНЖ, задолженности, получение карты, типовые проблемы.

Ценность: в чате накоплен большой пласт «народного» опыта (гайды, кейсы-таймлайны,
разбор ошибок), разбросанного по тысячам сообщений. Вики собирает это в
структурированный, перелинкованный и проверяемый по источникам справочник.

## Видение

- Вики — набор **кросс-линкованных Markdown-файлов** в директории `wiki/`.
- Каждое утверждение по возможности сопровождается **ссылкой на сообщение-источник**
  в чате, чтобы читатель мог проверить контекст и актуальность.
- Тон — нейтральный, практический. Вики не заменяет официальные разъяснения AIMA;
  информация получена опытным путём сообществом и может устаревать/содержать ошибки
  (обязательный дисклеймер в `wiki/README.md`).
- Структура тем — см. раздел «Категории» ниже (уточняется).

## Соглашения (согласовано с пользователем)

- **Аудитория вики — те, кто продлевается через новый портал** (ВНЖ истекает
  после 30.06.2025). Именно они составляют поток вновь приходящих с проблемами.
  Основной источник материала — топик `43114` «После 30.06.2025», он самый
  концентрированный и актуальный.
- **Но материал берём из любых топиков, если он релевантен этой аудитории.**
  Раннее решение «только 43114» оказалось некорректным: часть нужного знания
  живёт в соседних ветках (например, пошаговая инструкция по deferimento tácito
  — в закрепе `43113` «До 30.06.2025», отказы в сертификате разбирались в
  `63969` и `83903`). Отсекать их означало терять содержание.
  Критерий включения — **полезность для продлевающихся через портал**, а не
  номер топика. Материал, специфичный для старого очного процесса
  (визиты, слоты IRN), в вики не тащим.
- Ссылка на сообщение из другого топика строится так же:
  `https://t.me/aimairn/<topic_id>/<msg_id>` — топик определяется по цепочке
  `reply_to_message_id` до корня (см. «Как работать с данными»).
- **Ссылки на сообщения:** формат `https://t.me/aimairn/43114/<id>`, где `<id>` —
  поле `id` сообщения. Проверено: последнее сообщение #155182 совпадает с реальной
  ссылкой. Диапазон id топика: 43114…155182; период: 2025-07-16 … 2026-07-17.
- **Без имён авторов.** В статьях указываем только ссылку на сообщение-источник,
  без указания, кто автор.
- **Язык вики:** русский (аудитория — русскоязычные иммигранты).
  Португальские термины давать в оригинале + перевод; вести `wiki/glossary.md`.

## Данные экспорта

Каталог `Чат продления ВНЖ/` содержит два экспорта Telegram (HTML + `result.json`):

- `Export "После 30.06.2025" 2026-07-17/result.json` — **только топик 43114**
  (35 145 сообщений). **Рабочий источник для вики.**
- `Export all 2026-07-17/result.json` — полный чат (146 003 сообщения,
  окт 2024 – июл 2026), со всеми форумными топиками. **Тоже рабочий источник** —
  нужное знание бывает и в соседних ветках (см. соглашения выше).

Предпочитать `result.json` (машиночитаемый) парсингу HTML.

### Как работать с данными

```bash
# Извлечь чистый JSONL из топика 43114 (id, дата, автор, reply, текст):
jq -c '.messages[] | select(.type=="message")
  | {id, date, from, from_id, reply: .reply_to_message_id,
     text: ([.text_entities[].text] | join(""))}' \
  'Export "После 30.06.2025" 2026-07-17/result.json' > topic.jsonl

# Прочитать конкретное сообщение:
jq -r 'select(.id==79933) | .text' topic.jsonl
```

Полезные сигналы в тексте: хештеги `#кейспортал`, `#статистика`, `#кейсaima`
(структурированные кейсы-таймлайны); длинные сообщения (>600 симв.) часто гайды.

#### Определение топика для сообщения из полного экспорта

В экспорте нет поля «топик». Но `reply_to_message_id` у сообщения верхнего уровня
внутри ветки указывает на **id топика**, поэтому топик находится подъёмом по
цепочке reply до id из карты топиков (ниже). Это нужно, чтобы построить рабочую
ссылку `t.me/aimairn/<topic>/<id>` на сообщение из соседней ветки.

Скрипт-помощник (поиск по регэкспу с фильтром по топику и длине, печатает готовые
ссылки) писался в скратчпаде как `q.py` — при следующем заходе проще
восстановить его заново по этому описанию, чем искать.

⚠️ Ссылки на сообщения **проверять программно** перед публикацией: сообщение
существует и его вычисленный корень совпадает с топиком в ссылке.

## Метод исследования тем

Категории выведены из данных, а не назначены произвольно. Сигналы важности:
- **Частота ключевых терминов** по всему топику (см. тематическую карту в переписке).
- **Реакции** — поле `reactions` есть в `result.json` (у ~12.7k сообщений топика).
  Топ по сумме реакций = что сообщество сочло самым ценным. Отсеивать чисто
  эмоциональные вехи («принесли карту 🎉»); ценность — в длинных гайдах с реакциями.
- **Хештеги** `#кейспортал`/`#статистика`/`#кейсaima` — структурированные кейсы.
- **Длинные сообщения** (>600 симв.) — часто развёрнутые гайды.

Реакции подсветили недооценённые темы: борьба с «вечным анализом», сертификат
молчаливого согласия, коллективные обращения, канонические списки документов по
воссоединению, инструменты/боты сообщества.

## Архитектура публикации (MkDocs Material)

Источник истины — **дерево Markdown в `wiki/`** (= `docs_dir`). Сайт собирается
**MkDocs + тема Material**; конфиг — `mkdocs.yml` в корне проекта. Навигация
задаётся явным `nav:` в `mkdocs.yml`.

Структура дерева (`wiki/`):
- `index.md` — главная
- `process/` — процесс продления (+ `dokumenty/` с подстраницами
  `zaverenie-pasporta.md`, `szhatie-pdf.md`)
- `prerequisites/` — задолженности, данные/адрес
- `problems/` — дозапрос, застревание (+ `molchalivoe-soglasie.md`), коллективные
- `special/` — воссоединение и дети
- `reference/` — инструменты, ресурсы, хронология, глоссарий
- в каждой папке `index.md` — страница-обзор раздела.

Ссылки между статьями — **обычный markdown** с относительными путями
(`[...](../problems/zastrevanie.md)`), НЕ Zim-синтаксис `[[...]]`. Проверять:
`mkdocs build --strict`.

### Описания страниц (meta description / превью в мессенджерах)

У **каждой** страницы в начале файла — YAML front matter с `description:`:

```markdown
---
description: "Оплата продления ВНЖ 181,50 €: как приходит референс…"
---

# Оплата
```

Оно идёт в `<meta name="description">` и в `og:description` — это текст, который
видно в превью ссылки в Telegram (основной канал распространения вики), Slack,
WhatsApp. Без него Material подставляет общий `site_description` из `mkdocs.yml`,
и все страницы в превью выглядят одинаково.

Правила: 120–160 символов (длиннее — обрежется), суть страницы, а не общие слова;
значение в двойных кавычках (внутри есть `:`). **Новая статья без `description`
— незаконченная статья.** Проверка, что нигде не забыли:

```bash
cd wiki && find . -name '*.md' ! -name '*.md.md' \
  -exec sh -c 'head -3 "$1" | grep -q "^description:" || echo "нет description: $1"' _ {} \;
```

og-теги (`og:title`, `og:description`, `og:url`, …) собираются в
`overrides/main.html`, блок `extrahead` — Material сам их не выводит.

Внешние ссылки: включён `pymdownx.magiclink` — голые `https://…` URL и почты
авто-линкуются. НО не внутри бэктиков (код) и не для доменов без схемы
(`github.com/…`). Для таких — явная ссылка `[текст](https://…)` или `<почта>`.

### Сборка и деплой

```bash
mkdocs build --strict     # проверка (нет битых ссылок/nav)
mkdocs serve              # локальный предпросмотр
mkdocs gh-deploy --force  # деплой в ветку gh-pages → GitHub Pages
```

`mkdocs-material` ставит **пользователь** (через apt: `sudo apt install
mkdocs-material`). **Не устанавливай инструменты сам — проси пользователя.**

### Совместимость с Zim (вторично, для локального чтения)

`wiki/` открывается и в **Zim** (`notebook.zim`, `home=index` → главная вики,
отдельной Zim-страницы НЕ заводим). Zim в экспериментальном markdown-режиме при
резолвинге ссылки `X.md` ищет `X.md.md`, поэтому локально рядом с каждым `.md`
лежит симлинк `*.md.md → *.md` (в `.gitignore`, в сборку MkDocs не идёт —
исключён через `exclude_docs`). Регенерация после добавления статей:
```bash
cd wiki && find . -name '*.md' ! -name '*.md.md' -print0 | \
  while IFS= read -r -d '' f; do ln -sf "$(basename "$f")" "$f.md"; done
```
